# Hearth Design Audit — 2026-03-22

Post-refactor audit after colorize → distill → delight → animate passes.

---

## Anti-Patterns Verdict: MOSTLY PASSING

The four design passes removed the core AI-slop fingerprints. What remains:

- ✅ Gradient text — removed from page titles; intentionally kept on nav wordmark (one instance, logo-level)
- ✅ Glassmorphism on cards — removed; solid warm surfaces throughout main content
- ✅ Universal card hover lift — removed; `a.card` lifts, `div.card` stays flat
- ✅ Spring/bounce easing — confirmed zero instances remaining
- ✅ Body ambient glow blobs — removed
- ✅ Cold blue palette — shifted to warm amber
- ⚠️ **Mobile nav bottom bar** still has `backdrop-filter: blur(20px)` — missed in distill pass
- ⚠️ **20+ `surface-glass` references remain** across entry cards, task items, journal entries, companion notes, bridge rows, room buttons — inconsistent cleanup
- ⚠️ `--shadow-glow` still defined and used on `.live-dot` and one other element (line 480)
- ⚠️ `transition: all` in 8 places — lazy transition that animates expensive layout properties

Overall: no longer reads as AI-generated. The warm palette, solid surfaces, and differentiated hover behaviors are landing. The remaining issues are technical debt from incomplete cleanup sweeps, not fundamental design problems.

## Critical Issues

### C1 — Login form references undefined CSS tokens
**Location:** `app/login/LoginForm.tsx` — inline styles
**Severity:** Critical
**Category:** Theming
**Description:** The login form uses `var(--bg)`, `var(--text)`, and `var(--radius)` in its inline styles. These variables are NOT defined in `:root`. The defined tokens are `--bg-color`, `--text-main`, and `--radius-md`. These styles will silently fall back to `initial` — meaning the input may render with no background color, wrong text color, or no border-radius.
**Impact:** Login page may visually break under the new warm palette since the token names don't resolve.
**Recommendation:** Replace `var(--bg)` → `var(--bg-color)`, `var(--text)` → `var(--text-main)`, `var(--radius)` → `var(--radius-md)` in LoginForm.tsx.
**Command:** `/harden`

### C2 — `outline: none` on login input removes focus indicator
**Location:** `app/login/LoginForm.tsx` — input inline style
**Severity:** Critical
**Category:** Accessibility — WCAG 2.4.7 (Focus Visible, Level AA)
**Description:** `outline: none` is set directly on the password input with no replacement focus style. The only page that requires interaction to access the dashboard has no keyboard focus indicator on its primary field.
**Impact:** Keyboard-only users cannot see where focus is during login.
**Recommendation:** Remove `outline: none` or replace with `outline: 2px solid var(--accent); outline-offset: 2px`.
**Command:** `/harden`

## High-Severity Issues

### H1 — Two `<nav>` elements without `aria-label`
**Location:** `components/Nav.tsx` — both `<nav>` elements
**Severity:** High
**Category:** Accessibility — WCAG 4.1.2 (Name, Role, Value, Level A)
**Description:** There are two navigation landmarks — sidebar (desktop) and bottom bar (mobile). Neither has an `aria-label`. Screen readers will announce "navigation" twice with no way to distinguish them.
**Impact:** AT users navigating by landmarks hear two identical "navigation" regions.
**Recommendation:** Add `aria-label="Main navigation"` to `.nav-sidebar` and `aria-label="Mobile navigation"` to `.nav-bottom`.
**Command:** `/harden`

### H2 — Nav Unicode symbols read aloud to screen readers
**Location:** `components/Nav.tsx` — all `<span className="nav-sym">` elements
**Severity:** High
**Category:** Accessibility — WCAG 1.1.1 (Non-text Content, Level A)
**Description:** The nav symbols (◈, ⌂, ♥, ↹, ◌, ☑, ↑, ≡, ⌕) are decorative — the adjacent label text already communicates the meaning. Without `aria-hidden="true"`, screen readers will read both "◈ Home" or may read the Unicode codepoint name aloud.
**Impact:** Screen reader users hear garbled or redundant descriptions for every nav item.
**Recommendation:** Add `aria-hidden="true"` to all `.nav-sym` spans.
**Command:** `/harden`

### H3 — Search overlay has no `aria-live` region for results
**Location:** `components/SearchOverlay.tsx`
**Severity:** High
**Category:** Accessibility — WCAG 4.1.3 (Status Messages, Level AA)
**Description:** The overlay has `role="dialog"` and an Escape handler (good), but search results update dynamically with no `aria-live` announcement. Screen reader users who open search and type a query receive no feedback that results have appeared.
**Impact:** Search is effectively unusable for screen reader users.
**Recommendation:** Add `<div aria-live="polite" aria-atomic="true" className="sr-only">` announcing result count when results update.
**Command:** `/harden`

### H4 — 20+ remaining `surface-glass` references
**Location:** `app/globals.css` — lines 312, 319, 327, 341, 393, 405, 559, 560, 614, 774, 822, 841, 850, 888, 1124, 1236 and more
**Severity:** High
**Category:** Theming / Anti-patterns
**Description:** The distill pass removed glass from cards, nav sidebar, companion mood cards, and entry items — but missed journal entries (`.journal-entry`), task items (`.task-item`), companion note cards (`.companion-note-card`), room buttons, bridge rows, routine pip rows, and the nav search button hover. These elements still use `var(--surface-glass)` which, while the hue is now warm, still produces semi-transparent surfaces with no practical benefit.
**Impact:** Inconsistency — some surfaces are solid, others are semi-transparent for no reason. On warm amber backgrounds the effect is subtle but present.
**Recommendation:** Global replace: `var(--surface-glass)` → `var(--card-bg)`, `var(--surface-glass-hover)` → `var(--surface-raised)` across all remaining instances. After replacement, assess whether `--surface-glass` and `--surface-glass-hover` tokens are still needed.
**Command:** `/distill`

### H5 — Mobile nav bottom bar still has `backdrop-filter`
**Location:** `app/globals.css` — lines 168-170 (`.nav-bottom` or equivalent)
**Severity:** High
**Category:** Anti-patterns / Performance
**Description:** The distill pass removed `backdrop-filter` from `.nav-sidebar` but the mobile bottom nav still has `backdrop-filter: blur(20px)`. This is the element at line 168. On a solid warm background it adds zero visual benefit and non-trivial GPU cost on mobile.
**Impact:** Unnecessary GPU compositing on the device type (mobile) where it hurts most.
**Recommendation:** Remove `backdrop-filter` and `-webkit-backdrop-filter` from the bottom nav. Replace `var(--surface-glass)` with `var(--surface-base)`.
**Command:** `/distill`

## Medium-Severity Issues

### M1 — `transition: all` in 8 places
**Location:** `app/globals.css` — lines 69, 71, 148, 317, 335, 375, 732, 1166
**Severity:** Medium
**Category:** Performance
**Description:** `transition: all` causes the browser to watch every CSS property for changes and animate any that shift. This includes layout properties (width, height, padding, margin, top, left) that trigger expensive layout passes. Only `transform` and `opacity` are GPU-composited.
**Impact:** Potential jank on low-powered devices during any state change on these elements.
**Recommendation:** Replace with explicit properties: e.g. `transition: border-color 0.2s var(--ease-out), background 0.2s var(--ease-out)`.
**Command:** `/animate`

### M2 — `--shadow-glow` still defined and actively used
**Location:** `app/globals.css` — line 34 (definition), line 480, line 1313
**Severity:** Medium
**Category:** Anti-patterns
**Description:** The glow shadow variable remains in use on `.live-dot` (expected — this is the ceremonial heartbeat) and on one other element at line 480 (needs investigation). Global `text-shadow` glow was removed from `a:hover` in distill, but the variable and some usages remain.
**Impact:** Line 480 usage may be an unconsidered glow leftover. The `.live-dot` glow is intentional.
**Recommendation:** Audit line 480 — if it's not the live-dot or a specifically intentional element, remove. Keep `.live-dot` glow (it's the heartbeat of an open session).
**Command:** `/distill`

### M3 — Home section animation `both` fill mode may clip content
**Location:** `app/globals.css` — `.home-presence-grid`, `.home-section`
**Severity:** Medium
**Category:** Performance / Correctness
**Description:** Both home sections use `animation: fadeIn 0.5s var(--ease-out) Ns both`. The `both` fill mode holds opacity at 0 before the delay fires. If the user's browser or network is slow, or JS is disabled, content may be invisible for 150-250ms. This is also not conditioned by `prefers-reduced-motion` in the animation definition itself (though the media query block at the end covers it).
**Impact:** Minor content flash/delay on slow connections. `prefers-reduced-motion` block correctly suppresses it.
**Recommendation:** Acceptable as-is given the `prefers-reduced-motion` override exists. Consider reducing delay to 0.1s/0.2s if users report content feeling slow.
**Command:** none — monitor

### M4 — `home-section` margin-bottom conflicts with animation class
**Location:** `app/globals.css`
**Severity:** Medium
**Category:** CSS specificity
**Description:** `.home-section` originally had `margin-bottom: 3rem` defined. The animate pass added `animation: fadeIn 0.5s both` to the same selector. If these were merged from separate rules, specificity may behave unexpectedly.
**Impact:** Low risk — just note for future maintenance.
**Recommendation:** Consolidate into a single `.home-section` rule block.
**Command:** `/distill`

### M5 — `notes-card` still uses left border accent pattern
**Location:** `app/globals.css` line 1322
**Severity:** Medium
**Category:** Anti-patterns
**Description:** `.notes-card { border-left: 3px solid var(--accent) }` — the left-border stripe pattern we removed from `.card-accent` (changed to `border-top`). The notes card is a specific component, not the generic card, so this may be intentional as a "letters" visual metaphor. But it's the one remaining instance of the left-stripe pattern.
**Impact:** Minor visual inconsistency if the convention is now "top border = accent."
**Recommendation:** Decision: either change to `border-top` for consistency, or document that notes/letters intentionally use left border as a "margin" metaphor.
**Command:** `/distill` or intentional exception — document in `.impeccable.md`

### M6 — Nav sym `⌕` (search symbol) is non-standard Unicode
**Location:** `components/Nav.tsx` line 63
**Severity:** Medium
**Category:** Cross-platform rendering
**Description:** `⌕` (U+2315, TELEPHONE RECORDER) is used as a search icon. It may render as a tofu box on some platforms/fonts. The Caveat/Quicksand font stack may not include it.
**Impact:** Search icon may be invisible or render as □ on some systems.
**Recommendation:** Use a more common symbol (⌘, 🔍) or switch to an SVG icon for the search trigger.
**Command:** `/harden`

### M7 — `--drevan` uses shorthand HSL without CSS var
**Location:** `app/globals.css` line 49
**Severity:** Medium
**Category:** Theming consistency
**Description:** `--drevan: hsl(250, 80%, 68%)` — this is correct and intentional (Drevan's identity color is violet, distinct from the new amber accent). However, unlike other tokens, it's a fixed value. If the site ever needs theme switching, this won't participate in that system.
**Impact:** None currently. Cosmetic consistency issue only.
**Recommendation:** Acceptable as-is. Document that companion identity colors are intentionally fixed and not theme-responsive.

### M8 — `transition` on `<a>` and `<button>` global selectors
**Location:** `app/globals.css` lines 69, 71
**Severity:** Medium
**Category:** Performance
**Description:** Global `a { transition: all 0.2s ease }` and `button { transition: all 0.2s ease }` apply to every anchor and button on the site. This is the root cause of several `transition: all` instances — components inherit this and then override partially.
**Impact:** Every link and button animates all properties on every state change.
**Recommendation:** Change to `transition: color 0.2s var(--ease-out), background 0.2s var(--ease-out), border-color 0.2s var(--ease-out), opacity 0.2s var(--ease-out)`.
**Command:** `/animate`

## Patterns & Systemic Issues

1. **`surface-glass` was used as the universal surface token** — 20+ uses remain despite the intent to move to solid surfaces. The cleanup pass targeted structural components but missed content components (task items, journal entries, companion notes, room buttons).

2. **`transition: all` is a global habit** — appears on the `a` and `button` base selectors and propagates throughout. Fixing the root selectors would cascade improvements across the codebase.

3. **Decorative Unicode is not `aria-hidden`** — the nav sym pattern is used throughout. This needs a systematic pass, not component-by-component fixes.

---

## Positive Findings

- **`prefers-reduced-motion` block** — now present and comprehensive. This was missing entirely before the animate pass.
- **Companion identity color hover** — `color-mix(in srgb, var(--c) 15%, transparent)` is elegant; identity-tied, zero class management overhead.
- **`a.card` vs `div.card` hover semantics** — using element type rather than modifier class is clean and self-documenting.
- **`@keyframes breathe`** — 4s slow pulse is the right read for "presence" vs "alert." Meaningful differentiation.
- **Open session ceremonial treatment** — `border-top: 2px solid var(--accent)` + warm tint + extra padding lands correctly. The presence card reads as the most important element without being loud about it.
- **Login form fundamentals** — `htmlFor`/`id` association, `type="password"`, `autoComplete="current-password"` are all correct.
- **Search overlay** — Escape handling, `role="dialog"`, keyboard-accessible. Missing `aria-live` but the shell is correct.
- **Companion color system** — four distinct identity colors with consistent application across pages is well-designed and meaningful.
- **No bounce/elastic easing** — zero instances. Clean.
